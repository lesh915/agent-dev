## 0. 들어가며

### 0.1 이 책을 쓴 이유

2025년을 기점으로 AI 코딩 도구의 사용 방식이 조용히 바뀌었습니다.

| 시대 | 방식 |
|------|------|
| 프롬프트 시대 | `"이 함수 고쳐줘."` 한 줄 입력, 한 줄 출력 |
| 컨텍스트 시대 | 파일 몇 개를 붙여넣고, 배경을 설명하고, 방향을 안내 |
| **에이전트 시대** | AI가 직접 파일을 읽고, 코드를 쓰고, 테스트를 실행하고, PR을 올림 |

그런데 문제가 생겼습니다. 에이전트는 강력하지만 컨벤션을 무시하고, 없던 파일을 만들고, 이미 결정한 아키텍처를 뒤집습니다. 우리는 규칙을 알려준 적이 없으니까요.

이 문제를 해결하는 개념이 **하네스 엔지니어링(Harness Engineering)** 입니다.

> 하네스는 에이전트를 더 똑똑하게 만드는 기술이 아닙니다.  
> **에이전트가 올바르게 행동할 수밖에 없는 환경**을 설계하는 기술입니다.

**이 책에서 다루는 것:**

| 주제 | 내용 |
|------|------|
| 개념 | 하네스가 무엇인지, 왜 필요한지 |
| 구성 요소 | 지시 문서, 아키텍처 제약, 피드백 루프, 지식 저장소 |
| 실전 구축 | 0에서 첫 하네스까지, 실제 설정 파일 포함 |
| 사례 연구 | OpenAI Codex 팀, Stripe, Hashimoto 실험 |
| 전망 | 하네스가 왜 경쟁 우위가 되는가 |

**이 책에서 다루지 않는 것:**
- 특정 AI 도구 사용법 (Cursor, Claude, Copilot 튜토리얼 아님)
- 프롬프트 엔지니어링 기법
- AI 모델 파인튜닝

## 1. 하네스 엔지니어링 등장 배경

AI 코딩 도구는 짧은 시간 안에 세 번 진화했습니다. 프롬프트, 컨텍스트, 그리고 에이전트. 각 단계는 단순히 기능이 추가된 것이 아니라 **AI와 협업하는 방식 자체**가 바뀌었습니다.

---

### 1.1 프롬프트, 컨텍스트, 하네스 엔지니어링

#### AI 엔지니어링 세 가지 시대

**1단계: 프롬프트 시대**

```
"이 함수 버그 고쳐줘."
```

- 입력 하나, 출력 하나
- AI는 질문에 답하는 도구
- 대화가 끝나면 아무것도 기억하지 않음
- **개발자의 설명 능력**이 중요

**2단계: 컨텍스트 시대**

```
[파일 3개 첨부] + "이 구조에서 인증 모듈 추가해줘."
```

- 파일과 배경을 함께 제공해 결과물의 질이 향상
- 세션이 끝나면 컨텍스트가 사라짐
- 매번 새롭게 **컨텍스트를 준비하는 비용**이 발생

**3단계: 에이전트 시대**

```
"인증 모듈 추가하고 테스트 작성 후 PR 올려줘."
```

- 에이전트가 스스로 파일을 탐색하고, 코드를 작성하고, 테스트를 실행
- 사람의 개입 없이 작업을 완료
- **에이전트가 올바른 방향으로 일하게 만드는 능력**이 중요

#### 시대별 차이

| 구분 | 프롬프트 | 컨텍스트 | 하네스 |
|------|----------|----------|--------|
| 범위 | 1회 입력 | 1세션 | 영구 시스템 |
| 지속성 | 없음 | 세션 내 | 프로젝트 전체 |
| 주체 | 개발자가 지시 | 개발자가 안내 | 환경이 제약 |
| 실수 대응 | 재입력 | 재설명 | 구조적 방지 |
| 핵심 기술 | 프롬프트 작성 | 컨텍스트 구성 | 하네스 설계 |

#### 핵심

> 프롬프트와 컨텍스트는 **입력**을 개선하는 방식입니다.  
> 하네스는 **환경**을 설계하는 방식입니다.

에이전트에게 매번 규칙을 설명(입력)하는 대신, 규칙을 어길 수 없는 구조(환경)를 만듭니다.

---

### 1.2 에이전트와 하네스 엔지니어링

#### 하네스 없이 에이전트를 쓰면 생기는 문제들

**① 코드 스타일 불일치 (Convention Drift)**
- 에이전트는 팀의 네이밍 규칙이나 파일 구조 규칙을 모름
- PR마다 스타일이 달라지고, 리뷰어가 형식 문제를 지적하는 데 시간을 낭비

**② 구조 무시**
- 팀이 정해둔 코드 구조를 모르기 때문에 건너뜀
- 모듈 간 경계를 무시한 의존성 발생
- 당장은 동작하지만, 나중에 고치기 어려운 코드

**③ 같은 실수 반복**
- 에이전트는 이전 대화를 기억하지 않음
- 규칙을 문서화하지 않으면 매번 같은 일이 반복

**④ 파일 누적 현상**
- `utils2.py`, `helpers_new.js`, `temp_fix.ts` 같은 파일이 쌓임
- 어느 것이 실제로 사용되는지 파악 불가

**⑤ 컨텍스트 불안 (Context Anxiety)**
- 에이전트가 처리할 수 있는 정보량의 한계에 가까워졌다고 판단하면 작업을 제대로 마무리하지 않음
- 테스트를 건너뛰거나, 예외 처리를 생략하거나, TODO 주석만 남기고 끝냄

#### 공통 원인

> **에이전트는 지시받은 것만 압니다. 말하지 않은 규칙은 존재하지 않습니다.**

#### 일반적인 대응 방식

| 대응 방식 | 설명 | 단점 |
|-----------|------|------|
| 재입력 | 매번 프롬프트에 규칙 추가 | 세션마다 반복해야 하고 누락이 생김 |
| 사후 수정 | 에이전트 결과를 사람이 검수하고 수정 | 자동화의 이점이 사라짐 |
| **하네스 설계** | **규칙을 구조와 파일로 영구화** | **없음** |

> **핵심:** 문제는 에이전트가 아니라 환경에 있습니다.

## 2. 하네스 엔지니어링 개념

### 2.1 하네스 엔지니어링이란?

#### 정의

**하네스 엔지니어링(Harness Engineering)** 이란, AI 코딩 에이전트가 올바르게 동작하도록 환경을 설계하는 기술입니다.

'하네스(harness)'는 말에 씌우는 마구(馬具)에서 온 단어입니다. 말의 힘을 억누르는 게 아니라, 원하는 방향으로 이끄는 장치입니다. 에이전트도 마찬가지로, 능력을 제한하는 것이 아니라 그 능력이 올바른 방향으로 발휘되도록 구조를 갖추는 것입니다.

#### 하네스의 구성

| 구성 요소 | 역할 | 예시 |
|-----------|------|------|
| 지시 문서 | 에이전트에게 규칙을 전달 | `AGENTS.md`, `CLAUDE.md` |
| 아키텍처 제약 | 구조적으로 잘못된 코드를 차단 | 린터, 디렉토리 규칙 |
| 피드백 루프 | 에이전트 행동을 실시간으로 교정 | 테스트, CI 자동화 |
| 지식 저장소 | 팀의 결정과 맥락을 축적 | `docs/` 디렉토리 |

#### 하네스가 아닌 것

| 혼동하기 쉬운 개념 | 차이점 |
|-------------------|--------|
| 프롬프트 엔지니어링 | 입력을 개선하는 기술. 하네스는 환경을 설계하는 기술 |
| 코딩 컨벤션 문서 | 사람을 위한 문서. 하네스는 에이전트가 읽고 따르는 구조 |
| CI/CD 파이프라인 | 하네스의 일부가 될 수 있지만, 하네스 전체는 아님 |

#### 핵심 원칙

> **에이전트를 바꾸는 게 아니라, 에이전트가 일하는 환경을 바꾼다.**

---

### 2.2 Hashimoto의 6단계 도입 여정

Mitchell Hashimoto(HashiCorp 공동 창업자)는 AI 도입 경험을 6단계로 정리했습니다.

| 단계 | 이름 | 설명 |
|------|------|------|
| 1단계 | 자동완성 | IDE에서 코드 한 줄을 제안받는 수준. AI가 도구로 느껴지는 단계 |
| 2단계 | 코드 생성 | "이 함수 만들어줘"처럼 명시적으로 요청하고 결과를 받음 |
| 3단계 | 인터랙티브 편집 | 파일을 열어두고 AI와 대화하며 함께 수정 |
| 4단계 | 단일 에이전트 | AI가 스스로 파일을 읽고, 코드를 작성하고, 테스트를 실행 |
| 5단계 | 병렬 에이전트 | 여러 에이전트가 동시에 서로 다른 작업을 처리 |
| 6단계 | 자율 에이전트 시스템 | 이슈 읽기→코드 작성→PR 올리기 전 과정을 자율적으로 수행 |

#### 단계별 병목

| 단계 | 핵심 변화 | 새로운 병목 |
|------|-----------|------------|
| 1–2단계 | AI가 제안, 사람이 판단 | 프롬프트 작성 능력 |
| 3단계 | 맥락을 가진 대화 | 컨텍스트 준비 비용 |
| 4단계 | AI가 스스로 실행 | 에이전트 방향 설정 |
| 5–6단계 | 자율·병렬 처리 | 일관성, 재발 방지 구조 |

#### 하네스가 필요해지는 시점

4단계부터 문제가 시작됩니다. 에이전트가 스스로 판단하고 실행하는 순간, 개발자의 암묵적 규칙은 더 이상 전달되지 않습니다.

> *"에이전트가 실수했을 때, 나는 실수를 고치는 데서 멈추지 않는다. 그 실수가 구조적으로 재발 불가능하도록 만든다."*
> — Mitchell Hashimoto

### 2.3 하네스의 4가지 구성요소란?

```
프로젝트 루트/
│
├── AGENTS.md          ← ① 지시 문서
├── .eslintrc          ← ② 아키텍처 제약
│
├── tests/             ← ③ 피드백 루프
│   └── ...
│
└── docs/              ← ④ 지식 저장소
    ├── decisions/
    └── conventions/
```

#### 4가지 요소의 관계

```
지시 문서    →  에이전트가 "어떻게 할지" 안다
아키텍처 제약 →  잘못된 방향으로 가면 막힌다
피드백 루프  →  결과가 맞는지 즉시 확인한다
지식 저장소  →  왜 이렇게 하는지 이유를 안다
```

## 3. 하네스 구성요소

### 3.1 지시문서

에이전트에게 규칙을 전달하는 문서입니다. `AGENTS.md` 또는 `CLAUDE.md` 같은 파일로, 에이전트가 작업을 시작하기 전에 읽는 매뉴얼입니다.

**포함 내용:**
- 코드 스타일, 네이밍 규칙
- 절대 건드리면 안 되는 파일·디렉토리
- PR 작성 방식, 커밋 메시지 형식

사람을 위한 README처럼 **에이전트를 위한 행동 지침**입니다.

---

### 3.2 아키텍처 제약

잘못된 코드를 구조적으로 차단하는 장치입니다. 린터, 타입 검사, 디렉토리 규칙처럼 코드가 저장되거나 병합되기 전에 자동으로 검사합니다.

**기능:**
- 허용되지 않는 import 경로 차단
- 코드 스타일 자동 교정
- 특정 패턴 사용 금지

에이전트가 규칙을 몰라도, 잘못된 코드는 통과하지 못합니다.

---

### 3.3 피드백 루프

에이전트 행동을 실시간으로 교정하는 장치입니다. Martin Fowler는 이를 두 가지로 분류합니다.

| 종류 | 역할 | 예시 |
|------|------|------|
| 가이드(Guide) | 올바른 방향을 안내 | 테스트, 예제 코드 |
| 센서(Sensor) | 잘못된 행동을 감지 | CI 실패, 린터 경고 |

피드백이 빠를수록 에이전트는 더 적은 비용으로 올바른 결과를 냅니다.

---

### 3.4 지식 저장소

사람의 결정과 맥락을 축적하는 저장소입니다. `docs/` 디렉토리에 사람이 내린 결정, 채택한 이유, 포기한 대안을 기록합니다.

**기록 내용:**
- 왜 이 라이브러리를 선택했는가
- 이전에 시도했다가 실패한 방법
- 특정 구조를 유지하는 이유

에이전트는 매 세션 백지로 시작합니다. 지식 저장소는 그 공백을 채웁니다.

---

### 3.5 가비지 컬렉션

#### 드리프트란?

하네스를 구축해도 시간이 지나면 흐트러집니다. 에이전트가 임시 파일을 남기고, 사용하지 않는 코드가 쌓이고, 지시 문서와 실제 코드가 어긋나기 시작하는 현상을 **드리프트(drift)** 라고 합니다.

가비지 컬렉션은 드리프트를 자동으로 감지하고 정리하는 장치입니다.

#### 드리프트의 세 가지 유형

| 유형 | 증상 | 예시 |
|------|------|------|
| 코드 드리프트 | 사용하지 않는 코드 누적 | 미사용 함수, 죽은 import |
| 문서 드리프트 | 코드와 문서가 어긋남 | AGENTS.md의 규칙이 실제와 다름 |
| 구조 드리프트 | 디렉토리 규칙 붕괴 | 임시 파일, 규칙 밖 경로 |

#### 미사용 코드 자동 감지 (Python)

```python
# pyproject.toml — vulture로 죽은 코드 감지
[tool.vulture]
min_confidence = 80
paths = ["src/"]
ignore_names = ["test_*", "setUp", "tearDown"]
```

```bash
# 실행
pip install vulture
vulture src/
# 출력: src/utils/helpers.py:24: unused function 'format_legacy_date' (80% confidence)
```

#### 구조 드리프트 감지

```python
# scripts/check_structure.py
FORBIDDEN_PATTERNS = [
    "temp_*.py", "*_new.py", "*_old.py", "*_backup.*", "*_fix.*",
]
```

#### 정기 실행 자동화 (GitHub Actions)

```yaml
# .github/workflows/housekeeping.yml
name: 하네스 상태 점검
on:
  schedule:
    - cron: '0 9 * * 1'   # 매주 월요일 오전 9시

jobs:
  drift-check:
    runs-on: ubuntu-latest
    steps:
      - name: 미사용 코드 감지
        run: vulture src/
      - name: 문서 드리프트 감지
        run: python scripts/check_docs_drift.py
      - name: 구조 드리프트 감지
        run: python scripts/check_structure.py
```

#### AGENTS.md에 가비지 컬렉션 규칙 추가

```markdown
# AGENTS.md

## 정리 규칙
- 임시 파일은 작업 완료 즉시 삭제한다
- `temp_`, `_new`, `_old`, `_backup` 이름의 파일을 만들지 않는다
- 사용하지 않는 import는 즉시 제거한다
- 작업 중 생성한 디버그용 코드는 PR 전에 삭제한다
```

> 하네스도 코드처럼 관리해야 합니다. 정기적으로 점검하지 않으면 하네스 자체가 드리프트됩니다. 가비지 컬렉션은 하네스를 건강하게 유지하는 면역 체계입니다.

```
하네스 엔지니어링 = AI 에이전트가 올바르게 동작하는 환경 설계

4가지 구성요소:
  ① 지시 문서    → 에이전트에게 규칙 전달 (AGENTS.md)

  ④ 지식 저장소  → 팀의 결정과 맥락 축적 (docs/)
  (+) 가비지 컬렉션 → 드리프트 방지 자동화

핵심 원칙:
  에이전트를 바꾸는 게 아니라,
  에이전트가 일하는 환경을 바꾼다.
```
