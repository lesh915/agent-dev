## 1. 개요

### 1.1 제품 한 줄 정의

> **HE-AMS**는 AI 에이전트가 조직의 규칙을 자동으로 준수하고, 일탈을 구조적으로 방지하며, 장기적으로 개선되는 환경을 관리하는 시스템입니다.

### 1.2 배경

AI 코딩 도구의 진화는 3단계를 거쳤습니다.

| 시대 | 방식 | 병목 |
|------|------|------|
| 프롬프트 시대 | 한 줄 입력 → 한 줄 출력 | 프롬프트 작성 능력 |
| 컨텍스트 시대 | 파일 + 배경 → 세션 단위 결과 | 컨텍스트 준비 비용 |
| **에이전트 시대** | 자율 실행 → 파일 탐색·코드 작성·PR 생성 | **일관성·재발 방지 구조** |

에이전트 시대에는 **에이전트가 올바르게 동작할 수밖에 없는 환경(하네스)** 을 설계하는 것이 핵심 역량입니다. 그러나 현재 대부분의 팀은 하네스를 수동으로 관리하거나, 아예 갖추지 않은 상태입니다.

HE-AMS는 이 하네스를 시스템으로 구축·운영·진화시키는 플랫폼입니다.

### 1.3 핵심 원칙

> **에이전트를 바꾸는 것이 아니라, 에이전트가 일하는 환경을 바꾼다.**

## 2. 문제 정의

### 2.1 에이전트 도입 시 발생하는 5가지 구조적 문제

| # | 문제 | 증상 | 근본 원인 |
|---|------|------|-----------|
| P1 | **컨벤션 드리프트** | PR마다 코드 스타일 불일치, 리뷰 비용 증가 | 팀 규칙이 에이전트에게 전달되지 않음 |
| P2 | **구조 무시** | 모듈 경계 붕괴, 예상치 못한 의존성 발생 | 아키텍처 제약이 에이전트에게 투명하지 않음 |
| P3 | **반복 실수** | 동일한 에러가 세션마다 재발 | 에이전트는 이전 대화를 기억하지 않음 |
| P4 | **파일 누적** | `temp_`, `_new`, `_backup` 파일이 쌓임 | 정리 규칙이 구조적으로 강제되지 않음 |
| P5 | **컨텍스트 불안** | 테스트 생략, TODO 주석으로 마무리 | 컨텍스트 한계 근처에서 에이전트가 타협 |

### 2.2 현재 팀의 대응 방식과 한계

| 대응 방식 | 설명 | 한계 |
|-----------|------|------|
| 재입력 (Re-prompting) | 매 세션마다 규칙을 프롬프트에 포함 | 누락 발생, 확장 불가 |
| 사후 수정 (Post-review) | 에이전트 결과를 사람이 검수 | 자동화 이점 소멸 |
| 단편 문서화 | README에 규칙 일부 기록 | 에이전트가 읽고 따르는 구조가 아님 |

### 2.3 규모별 심각도

```
단일 에이전트 (Hashimoto 4단계)
  → 방향 설정 문제. 팀이 수동으로 관리 가능.

병렬 에이전트 (5단계)
  → 일관성 붕괴. 여러 에이전트가 서로 다른 판단을 내림.

자율 에이전트 시스템 (6단계)
  → 재발 방지 구조 없이는 운영 불가. 사람의 개입 없이 실수가 축적됨.
```

## 3. 목표 및 성공 기준

### 3.1 제품 목표

| 목표 | 설명 |
|------|------|
| **G1. 일관성 보장** | 모든 에이전트가 동일한 규칙·스타일·구조를 따르도록 환경을 구성 |
| **G2. 재발 방지** | 에이전트의 실수가 구조적으로 다시 발생하지 않는 메커니즘 제공 |
| **G3. 하네스 가시성** | 현재 하네스의 상태(드리프트·누락·충돌)를 실시간으로 파악 |
| **G4. 진화 가능성** | 팀의 학습과 결정이 하네스에 자동으로 축적되는 피드백 루프 제공 |
| **G5. 낮은 도입 비용** | 기존 개발 환경(Git, CI, IDE)에 침습 최소화로 통합 |

### 3.2 성공 지표 (KPI)

| 지표 | 현재 (기준선) | 목표 (6개월) |
|------|--------------|-------------|
| 에이전트 PR의 컨벤션 위반율 | 측정 안 됨 | ≤ 5% |
| 동일 실수 재발 간격 | 즉시 재발 | 구조적 차단으로 재발 0 |
| 하네스 문서 vs 실제 코드 일치율 | 측정 안 됨 | ≥ 90% |
| 개발자가 에이전트 결과를 수정하는 비율 | 측정 안 됨 | ≤ 20% |
| 하네스 드리프트 감지에서 해소까지 소요 시간 | 발견 불가 | ≤ 24시간 |

## 4. 사용자 및 이해관계자

### 4.1 주요 사용자

#### 페르소나 1: 하네스 설계자 (Harness Architect)
- **역할:** 팀의 에이전트 운영 환경을 설계하고 유지하는 시니어 엔지니어
- **Pain Point:** 하네스 규칙이 여러 파일에 흩어져 있어 전체 상태 파악이 어려움
- **핵심 니즈:** 하네스의 일관성 검증, 드리프트 감지, 규칙 적용 자동화

#### 페르소나 2: 에이전트 사용자 (Agent User)
- **역할:** 일상적인 개발 작업에 에이전트를 활용하는 개발자
- **Pain Point:** 에이전트 결과가 팀 규칙을 어겨 리뷰에서 반려됨
- **핵심 니즈:** 에이전트가 처음부터 올바른 결과를 내놓을 것

#### 페르소나 3: 엔지니어링 매니저 (Engineering Manager)
- **역할:** 팀의 에이전트 도입 성과를 추적하고 의사결정을 내리는 관리자
- **Pain Point:** 에이전트 도입 효과가 보이지 않음, 문제 발생 시 원인 파악 어려움
- **핵심 니즈:** 에이전트 품질 지표 대시보드, 하네스 ROI 측정

### 4.2 이해관계자

| 역할 | 관심사 |
|------|--------|
| 보안팀 | 에이전트가 민감 파일에 접근하지 않는가 |
| DevOps | 기존 CI/CD 파이프라인과의 통합 |
| 제품팀 | 에이전트 도입으로 인한 개발 속도 향상 |

         ↕                    ↕                    ↕
   Git / GitHub          CI / CD 파이프라인      IDE / 에디터

### 5.2 하네스 계층 구조

```
글로벌 하네스 (조직 공통 규칙)
    └── 팀 하네스 (팀별 규칙)
            └── 프로젝트 하네스 (프로젝트별 규칙)
                    └── 태스크 하네스 (작업별 임시 규칙)
```

상위 계층의 규칙은 하위 계층에서 재정의할 수 없습니다 (보안·컨벤션 관련 항목 한정).

### 5.3 설계 원칙

| 원칙 | 내용 |
|------|------|
| **구조 우선** | 규칙은 문서가 아닌 시스템이 강제한다 |
| **최소 침습** | 기존 Git·CI 워크플로우를 변경하지 않는다 |
| **관찰 가능성** | 모든 에이전트 행동은 추적·분석 가능하다 |
| **점진적 도입** | Hashimoto 4단계부터 6단계까지 단계별 기능 활성화 |
| **자기 진화** | 하네스가 에이전트의 실수로부터 자동으로 학습한다 |

## 6. 핵심 기능 요구사항

### 6.1 하네스 관리 (Harness Management)

#### F1. 지시문서 관리 시스템

**목적:** `AGENTS.md` / `CLAUDE.md` 등 에이전트 지시문서를 중앙에서 관리하고 배포.

| 기능 | 설명 | 우선순위 |
|------|------|----------|
| F1-1 | 지시문서 생성·편집·버전 관리 (Git 기반) | P0 |
| F1-2 | 계층별 지시문서 상속 및 오버라이드 | P0 |
| F1-3 | 지시문서 → 대상 저장소 자동 동기화 | P1 |
| F1-4 | 지시문서 항목별 에이전트 준수율 추적 | P1 |
| F1-5 | 지시문서 변경 이력 및 변경 이유 기록 | P2 |

**지시문서 표준 구조:**

```markdown
# AGENTS.md

## 코드 스타일
## 디렉토리 규칙
## 절대 금지 항목
## PR 작성 방식
## 알려진 함정 (Known Pitfalls)
## 정리 규칙 (가비지 컬렉션)
```

#### F2. 아키텍처 제약 레지스트리

**목적:** 린터·타입 검사·import 규칙 등 구조적 제약을 중앙에서 등록하고 강제.

| 기능 | 설명 | 우선순위 |
|------|------|----------|
| F2-1 | 제약 규칙 등록 (허용/금지 패턴, import 규칙) | P0 |
| F2-2 | Git pre-commit 훅 자동 생성 및 배포 | P0 |
| F2-3 | CI 단계별 제약 검사 통합 | P1 |
| F2-4 | 제약 위반 시 에이전트에게 구체적인 수정 힌트 반환 | P1 |
| F2-5 | 제약 규칙의 예외 관리 (화이트리스트) | P2 |

#### F3. 지식 저장소 관리

**목적:** 팀의 의사결정, 실패 경험, 맥락을 에이전트가 참조할 수 있는 형태로 축적.

| 기능 | 설명 | 우선순위 |
|------|------|----------|
| F3-1 | 의사결정 기록 (ADR: Architecture Decision Record) 템플릿 및 관리 | P0 |
| F3-2 | 에이전트 실수 → 지식 저장소 자동 변환 워크플로우 | P1 |
| F3-3 | 지식 항목의 검색 및 에이전트 컨텍스트 주입 | P1 |
| F3-4 | 지식 항목의 유효성 만료 관리 (stale 감지) | P2 |

### 6.2 에이전트 실행 관리 (Agent Execution Management)

#### F4. 에이전트 오케스트레이션

**목적:** 단일·병렬·자율 에이전트의 작업을 스케줄링하고 모니터링.

| 기능 | 설명 | 우선순위 |
|------|------|----------|
| F4-1 | 작업 큐 및 에이전트 할당 | P0 |
| F4-2 | 병렬 에이전트 실행 및 충돌 감지 | P1 |
| F4-3 | 에이전트 작업 상태 실시간 모니터링 | P1 |
| F4-4 | 작업 실패 시 자동 재시도 및 에스컬레이션 | P2 |

#### F5. 피드백 루프 자동화

**목적:** 에이전트 행동을 실시간으로 교정하는 가이드(Guide)와 센서(Sensor) 구성.

| 종류 | 기능 | 우선순위 |
|------|------|----------|
| **가이드** | 테스트 기반 방향 안내 | P0 |
| **가이드** | 예제 코드 주입 (지식 저장소 연동) | P1 |
| **센서** | CI 실패 → 에이전트에게 피드백 자동 반환 | P0 |
| **센서** | 린터 경고 → 에이전트 재작업 트리거 | P1 |
| **센서** | PR 리뷰 패턴 분석 → 지시문서 업데이트 제안 | P2 |

### 6.3 드리프트 감지 및 가비지 컬렉션

#### F6. 드리프트 감지 시스템

**목적:** 하네스와 실제 코드베이스 간의 괴리를 자동으로 감지.

| 드리프트 유형 | 감지 방법 | 우선순위 |
|--------------|-----------|----------|
| **코드 드리프트** | 미사용 함수·import 탐지 (vulture, knip 등) | P0 |
| **문서 드리프트** | AGENTS.md 규칙 vs 실제 코드 패턴 비교 | P1 |
| **구조 드리프트** | 금지 파일명 패턴 탐지 (`temp_*`, `*_backup`) | P0 |
| **하네스 드리프트** | 지시문서 규칙이 실제 동작과 불일치 탐지 | P2 |

#### F7. 가비지 컬렉션 자동화

| 기능 | 설명 | 우선순위 |
|------|------|----------|
| F7-1 | 정기 드리프트 스캔 스케줄링 (주 단위 기본) | P0 |
| F7-2 | 드리프트 리포트 생성 및 담당자 알림 | P0 |
| F7-3 | 안전한 항목 자동 정리 (미사용 import 등) | P1 |
| F7-4 | 정리 이력 및 복구 지점 관리 | P2 |

### 6.4 관찰 및 분석 (Observability)

#### F8. 하네스 상태 대시보드

| 지표 | 설명 | 우선순위 |
|------|------|----------|

## 8. 하네스 구성요소별 설계 원칙

### 8.1 지시문서 (Instruction Documents) 설계 원칙

```
원칙 1: 목차형 구성
  → 에이전트는 전체를 읽지 않을 수 있다. 섹션별로 독립적으로 참조 가능해야 한다.

원칙 2: 금지 > 허용
  → "X를 하라"보다 "Y를 하지 마라"가 에이전트에게 더 명확하게 전달된다.

원칙 3: 예시 포함 의무화
  → 규칙마다 올바른 예시 1개, 잘못된 예시 1개를 포함한다.

원칙 4: Known Pitfalls 섹션 필수
  → 에이전트가 반복한 실수는 즉시 이 섹션에 추가한다.

원칙 5: 버전 관리
  → 지시문서는 코드처럼 Git으로 관리하고, 변경 이유를 커밋 메시지에 기록한다.
```

### 8.2 아키텍처 제약 (Architecture Constraints) 설계 원칙

```
원칙 1: 사전 차단 우선
  → 잘못된 코드는 저장 전에 차단한다 (pre-commit > CI > PR 리뷰 순서).

원칙 2: 에러 메시지에 수정 힌트 포함
  → "규칙 위반"이 아닌 "X 대신 Y를 사용하세요"로 안내한다.

원칙 3: 자동화 가능한 것은 자동화
  → 린터로 자동 수정 가능한 항목은 수동 확인 없이 자동 적용한다.

원칙 4: 제약의 이유를 코드에 기록
  → 제약 규칙 파일에 왜 이 제약이 있는지 주석으로 명시한다.
```

### 8.3 피드백 루프 (Feedback Loops) 설계 원칙

```
원칙 1: 빠른 피드백 우선
  → 피드백 지연 = 에이전트 비용 증가. pre-commit이 CI보다 항상 먼저.

원칙 2: 가이드와 센서 균형
  → 센서(금지)만 있으면 에이전트가 막히고, 가이드(안내)만 있으면 강제력이 없다.

원칙 3: 피드백의 실행 가능성
  → 에이전트가 피드백을 받으면 즉시 수정할 수 있도록 구체적인 정보를 제공한다.

원칙 4: 피드백 기록 및 분석
  → 같은 피드백이 반복된다면 하네스 자체를 개선해야 한다는 신호다.
```

### 8.4 지식 저장소 (Knowledge Repository) 설계 원칙

```
원칙 1: WHY 중심 기록
  → 무엇을 했는지(코드)는 이미 있다. 왜 이렇게 결정했는지를 기록한다.

원칙 2: 실패 사례 필수 포함
  → 시도했다가 포기한 방법을 반드시 기록한다. 에이전트가 같은 함정에 빠지지 않도록.

원칙 3: 유효 기간 관리
  → 지식은 낡는다. 6개월마다 유효성 검토를 의무화한다.

원칙 4: 에이전트 접근 최적화
  → 지식 항목은 에이전트 컨텍스트에 주입할 수 있는 크기(~500토큰)로 관리한다.
```

### 8.5 가비지 컬렉션 (Garbage Collection) 설계 원칙

```
원칙 1: 정기 실행 의무화
  → 드리프트 감지는 선택이 아닌 기본 설정. 주 1회 이상.

원칙 2: 감지와 정리를 분리
  → 자동 감지 후 정리는 사람이 승인하거나, 안전하다고 증명된 경우만 자동화.

원칙 3: 하네스 자체도 GC 대상
  → 지시문서·제약 규칙·지식 저장소도 정기적으로 정리한다.

원칙 4: 정리 이력 보존
  → 무엇을 왜 정리했는지 기록한다. 언제든 복구 가능해야 한다.
```

## 10. 리스크 및 제약사항

### 10.1 기술 리스크

| 리스크 | 발생 가능성 | 영향도 | 완화 방안 |
|--------|------------|--------|-----------|
| 에이전트 API 변경으로 인한 연동 중단 | 높음 | 높음 | 어댑터 패턴으로 API 격리, 버전별 테스트 |
| 드리프트 감지 허위 양성 증가 | 중간 | 중간 | 신뢰도 임계값 조정 기능, 화이트리스트 관리 |
| 대규모 코드베이스에서 스캔 성능 저하 | 중간 | 낮음 | 증분 스캔(incremental scan) 구현 |
| 병렬 에이전트 간 하네스 규칙 충돌 | 낮음 | 높음 | 잠금(lock) 메커니즘, 에이전트 격리 실행 |

### 10.2 조직 리스크

| 리스크 | 발생 가능성 | 영향도 | 완화 방안 |
|--------|------------|--------|-----------|
| 개발자가 하네스를 우회하려는 시도 | 높음 | 높음 | 제약 우회 로그 기록, 팀 문화 교육 |
| 하네스 설계자 부재 시 시스템 퇴화 | 중간 | 높음 | 하네스 건강도 자동 모니터링 및 알림 |
| 지나친 제약으로 에이전트 효용 저하 | 중간 | 중간 | 제약 적용 전 영향 분석 기능 제공 |

### 10.3 범위 제약

- 이 시스템은 에이전트를 더 똑똑하게 만들지 않습니다. **에이전트가 올바르게 행동할 수밖에 없는 환경**을 제공합니다.
- 특정 AI 모델(Claude, GPT-4 등)에 종속되지 않도록 설계합니다.
- 프롬프트 엔지니어링 기법 및 모델 파인튜닝은 범위 밖입니다.
